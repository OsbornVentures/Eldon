# How ELDON Works

## Installing from scratch (agent-readable)

SETUP.md covers the steps. Three things that doc doesn't spell out explicitly:

**The model filename matters.** llama-server is started with
`--jinja --chat-template-file templates/gemma4.jinja`. That flag is what enables
Gemma 4's native tool-calling token format. Without it llama-server falls back to a
generic format and all tool calls break silently. The launcher (`launcher/eldon.ps1`)
adds this flag automatically — if you're on Linux/macOS and launching manually, it is
the one flag you cannot omit.

**The template file path is relative to the eldon repo root.** It must exist at
`templates/gemma4.jinja` when llama-server starts. The launcher resolves this from
`$PSScriptRoot` so it just works from the repo directory.

**Port split is intentional.** In agent mode, llama-server listens on `:8081` and the
proxy on `:8080`. Point clients at `:8080`, never `:8081` directly — the proxy is the
tool engine. In API-only mode (no proxy), llama-server runs on `:8080` and you talk to
it directly.

## The Gemma 4 + llama.cpp tool calling stack

This combination is non-obvious enough to document properly.

### Why Gemma 4's tool calling is different

Gemma 4 uses a token-native tool call format that is completely unlike GPT-style
`{"name":..., "arguments":...}` JSON blobs. The model was trained to emit and parse
structured tokens:

```
<|tool_call>call:tool_name{arg:value}<tool_call|>
<|tool_response>response:tool_name{result:value}<tool_response|>
<|channel>thought\n...\n<channel|>
```

These are not post-hoc wrappers — they are the format the weights were trained on.
If the prompt does not use these tokens in the right positions, the model does not
reliably produce structured output regardless of instructions.

### The role of gemma4.jinja

`templates/gemma4.jinja` is a Jinja2 chat template that llama-server applies **before
every inference call**, at the C++ layer, zero Python overhead. It takes the
OpenAI-format message array (system / user / assistant / tool roles) and renders it
into the raw token sequence Gemma 4 expects.

Key things it handles:

- Wraps each conversational turn in `<|turn>role\n` ... `<turn|>` markers
- Renders every tool definition into the system turn as a
  `<|tool>declaration:name{description:"...",parameters:{...}}<tool|>` block —
  Gemma 4's native schema format, not JSON Schema
- Converts `finish_reason: tool_calls` assistant messages into
  `<|tool_call>call:name{args}<tool_call|>` tokens
- Wraps tool result messages in `<|tool_response>response:name{...}<tool_response|>`
- Strips thinking blocks (`<|channel>thought...<channel|>`) from assistant history so
  they don't inflate the context on subsequent turns
- Handles multi-turn continuation — when an assistant turn is followed by more
  assistant content (tool result → next reply) it avoids double-inserting the turn
  header

As of mid-2025, llama.cpp has no built-in Gemma 4 template. Every other ELDON-style
setup guide for Gemma 4 that omits this template gets broken or no tool calling.
This file is the reason ELDON's tool calling actually works.

### The grammar (loop path only)

`runtime/grammar.gbnf` is a GBNF grammar compiled by llama-server into a finite-state
machine applied at the **logit level** during token sampling on the raw `/completion`
path. It enforces this exact structure:

```
<tool>TOOL_NAME</tool>
<args>{"key": "value"}</args>
<reason>brief explanation</reason>
```

`TOOL_NAME` is a disjunction of every registered tool name, auto-generated by
`tools/_meta.py` on each `loop.py` startup. The model **cannot** emit an unregistered
tool name — the grammar physically prevents it before sampling, not as a filter after.

This eliminates an entire class of failure mode: the model can hallucinate plausible
tool names when unconstrained, especially mid-conversation. With the grammar, that is
impossible by construction.

The grammar is **not used on the proxy path** — the proxy uses the chat completions
API where llama-server handles structure via the jinja template.

## Proxy path vs scheduler/loop path

ELDON has two distinct execution paths with different tradeoffs.

### Proxy path (`proxy.py`)

```
Client → :8080 (proxy.py) → :8081 (llama-server)
```

The proxy is a Flask app. Every request except `/v1/chat/completions` is forwarded
transparently (the llama-server browser UI works through it unchanged, with a URL
linkifier injected into the HTML).

For `/v1/chat/completions` the proxy runs an agentic loop:

1. Client sends a normal OpenAI chat request to `:8080`
2. Proxy injects the full TOOL_DEFS list if the client didn't supply tools
3. Proxy calls llama-server at `:8081` with `stream: false` (always, for tool turns)
4. llama-server applies `gemma4.jinja` to format the message array into Gemma 4 tokens,
   runs inference, returns a response
5. If `finish_reason == "tool_calls"`, the proxy:
   - Appends the assistant message (with tool_calls) to the conversation
   - Executes each requested tool via `_dispatch()` (Python, local)
   - Appends `role: tool` result messages to the conversation
   - Goes back to step 3
6. When the model returns a normal text response (no tool_calls), the proxy re-issues
   that final request with `stream: true` and pipes the SSE stream back to the client
7. The client receives a single streaming response as if no tool calls happened

The client sees one response. The proxy may have called llama-server 2–20 times
internally. Max turns defaults to 20, configurable per-request via `max_tool_turns`.

This path is used by: the browser UI (llama-server's built-in client), any
OpenAI-compatible API client pointed at `:8080`, and interactive sessions.

### Scheduler/loop path (`scheduler.py` → `loop.py`)

```
scheduler.py  →  loop.loop()  →  llama-server :8081/completion (raw)
```

The scheduler polls `runtime/scheduled_tasks.json` every 60 seconds. When a task's
`next_run` timestamp is due, it calls `loop.loop(prompt)` directly — no proxy, no Flask,
no HTTP client involved on the receiving end.

`loop.py` uses the raw `/completion` endpoint (not chat completions). This means:

- It manually wraps the prompt in `<start_of_turn>user\n...\n<end_of_turn>\n<start_of_turn>model\n`
  (Gemma 2/3 legacy markers, the only format llama.cpp reliably processes on the raw
  completion path as of mid-2025)
- It sends `grammar: <contents of grammar.gbnf>` in the payload, enabling grammar-
  constrained sampling
- It parses the structured response itself: `<tool>`, `<args>`, `<reason>` tags via regex
- It dispatches the tool call, appends to `STATE.md`, and loops

The loop also handles:
- **Tool hot-reload**: if `skill_make` succeeds, `_meta.regenerate_all()` rebuilds
  `TOOLS.md` and `grammar.gbnf` immediately, and `load_tools()` re-imports the new
  module — the grammar update takes effect on the next turn
- **Stuck detection**: inspects the last 6 STATE.md lines for repeated tool+args
  triplets (≥3x), tool name repetition (≥5x), or repeated identical errors
- **State trimming**: keeps STATE.md under 100 lines via FIFO with a summary line
  recording tool-call counts for the archived batch

The loop path is stateless from llama-server's perspective — each turn is a fresh
`/completion` request with the full context reconstructed by `compose_context()`.
The "memory" between turns lives in `STATE.md`, `OPEN_LOOPS.md`, and ChromaDB, not
in llama-server's KV cache.

### Why two paths

| | Proxy path | Loop path |
|---|---|---|
| Entry point | HTTP client | scheduler or CLI |
| LLM endpoint | `/v1/chat/completions` | `/completion` |
| Tool format | Gemma 4 native via jinja | grammar.gbnf XML tags |
| Context handling | llama-server manages | `compose_context()` rebuilds each turn |
| Streaming | SSE to client | none (blocking) |
| Use case | Interactive, browser, API | Scheduled jobs, autonomous tasks |

The proxy gives you streaming and interactive UX. The loop path gives you cron-driven
automation and grammar-enforced reliability. They share the same tool registry
(`tools/core/` and `tools/learned/`) and the same `STATE.md` log.

## The full token flow for one tool call (proxy path)

```
User: "what's the weather in Austin?"

1. proxy receives POST /v1/chat/completions
   messages: [{role: user, content: "what's the weather in Austin?"}]
   tools: [...all TOOL_DEFS injected by proxy...]

2. proxy POST to llama-server /v1/chat/completions
   llama-server applies gemma4.jinja:
     <|turn>system
     <|tool>declaration:weather{description:"...",parameters:{...}}<tool|>
     ... (all other tool declarations)
     <turn|>
     <|turn>user
     what's the weather in Austin?
     <turn|>
     <|turn>model
   inference runs → model emits:
     <|tool_call>call:weather{location:"Austin, TX"}<tool_call|>
   llama-server returns finish_reason: "tool_calls"

3. proxy dispatches: weather({"location": "Austin, TX"})
   → calls wttr.in API → returns {temp: "82°F", ...}

4. proxy appends to messages:
   {role: assistant, tool_calls: [{function: {name: "weather", ...}}]}
   {role: tool, tool_call_id: "...", content: '{"temp":"82°F",...}'}

5. proxy POST to llama-server /v1/chat/completions again
   gemma4.jinja renders tool result as:
     <|tool_response>response:weather{temp:"82°F",...}<tool_response|>
   model generates text answer: "It's currently 82°F in Austin..."
   finish_reason: "stop"

6. proxy re-POSTs with stream: true, pipes SSE back to client
   client receives: "It's currently 82°F in Austin..."
```

One user request → two llama-server calls → one streamed response. No tool calls
visible to the client.
