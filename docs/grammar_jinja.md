# Grammar and Template

ELDON uses two files that together define how Gemma 4 structures its tool calls.

## gemma4.jinja

A Jinja2 chat template processed by llama-server before every inference call. It formats the conversation history (system, user, assistant, tool messages) into the raw token sequence Gemma 4 expects.

Key responsibilities:
- Wraps turns in Gemma 4's `<|turn>role` / `<turn|>` markers.
- Renders tool definitions into the system turn using Gemma 4's native declaration format (`<|tool>declaration:name{...}<tool|>`).
- Renders tool calls as `<|tool_call>call:name{args}<tool_call|>`.
- Renders tool responses as `<|tool_response>response:name{...}<tool_response|>`.
- Strips thinking blocks (`<|channel>thought...<channel|>`) from assistant content when building history.

The template is applied by llama-server, not Python. It does not contribute to the Python process overhead and is rendered once per request at the C++ layer.

## grammar.gbnf

A GBNF (GGML BNF) grammar file that constrains llama-server's output token sampling. It is compiled into a state machine by llama-server at load or first use and applied at the logit level during generation — not as a post-processing filter.

The grammar enforces this output structure for the **loop.py path** (direct completion, not chat completions):

```
<tool>TOOL_NAME</tool>
<args>{"key": "value"}</args>
<reason>brief explanation</reason>
```

`tool-name` is an enumeration of every registered tool name. This means the model cannot output an invalid tool name — the grammar physically prevents it.

`grammar.gbnf` is auto-generated by `tools/_meta.py` from the current tool set. `grammar.template` holds the structure with `__TOOL_NAMES__` as a placeholder. Do not edit `grammar.gbnf` by hand; run `python -c "from tools import _meta; _meta.regenerate_all()"` to rebuild.

## What the grammar does not do

The grammar constrains the **loop.py path** only. The proxy path (`proxy.py`) uses the OpenAI chat completions API with structured tool definitions — llama-server handles tool call format internally via the jinja template. The gbnf grammar is not used in the proxy path.

The two paths also differ in how they wrap prompts with chat control tokens. The proxy path uses native Gemma 4 tokens (`<|turn>` / `<turn|>`) applied by llama-server via `gemma4.jinja`. The loop path uses legacy Gemma 2/3 markers (`<start_of_turn>` / `<end_of_turn>`) as a llama.cpp compatibility fallback — current llama.cpp builds lack a dedicated Gemma 4 chat template and the new tokens are not reliably recognized on the raw `/completion` endpoint. See `docs/loop.md` for the full rationale.

## Token cost

`grammar.gbnf` is not part of the prompt token stream. It is a parameter to the `/completion` endpoint. Its size does not affect context window usage.

`gemma4.jinja` produces the system turn, which IS tokenized. The rendered system turn (tool definitions + any system prompt) consumes context tokens and is cached by llama-server's KV cache between requests with identical prefixes.
